# 影片 Resize 工具（網頁版）— 開發問題與教訓彙整

本文記錄開發過程中遇到的所有問題、根本原因、解決方式，以及未來如何避免。

---

## 問題一：FFmpeg Worker 跨來源錯誤（最關鍵）

### 錯誤訊息

```
Failed to construct 'Worker': Script at
'https://esm.sh/@ffmpeg/ffmpeg@0.12.10/es2022/worker.js'
cannot be accessed from origin 'https://aqualung929.github.io'.
```

### 症狀

頁面載入後馬上跳出「FFmpeg 載入失敗」，無法使用。

### 根本原因

`@ffmpeg/ffmpeg@0.12.x` 的原始碼（`classes.js`）用以下方式建立 Web Worker：

```javascript
new Worker(new URL("./worker.js", import.meta.url), { type: "module" })
```

`import.meta.url` 是 ES Module 的特殊變數，指向「這個 JS 檔案本身的 URL」。
當從 esm.sh CDN 載入時：

```
import.meta.url = https://esm.sh/@ffmpeg/ffmpeg@0.12.10/es2022/classes.js
↓ 解析 "./worker.js"
Worker URL  = https://esm.sh/@ffmpeg/ffmpeg@0.12.10/es2022/worker.js  ← 跨來源！
```

瀏覽器安全政策規定：**Web Worker 只能從同來源（same-origin）的 URL 建立**，跨來源直接拒絕，不嘗試任何網路請求。

### 嘗試過但無效的解法

**嘗試 1：加入 `workerURL` 到 `ffmpeg.load()`**

```javascript
await ffmpeg.load({
  coreURL: await toBlobURL(...),
  wasmURL: await toBlobURL(...),
  workerURL: await toBlobURL('https://esm.sh/.../worker.js', 'text/javascript'),  // ❌ 無效
});
```

**失敗原因：** `ffmpeg.load()` 的 `workerURL` 參數是傳給 WASM 核心用的（@ffmpeg/core 的 worker），**不是**控制 `@ffmpeg/ffmpeg` 自身的 class worker URL。class worker 的 URL 在 `classes.js` 內部以 `import.meta.url` 硬計算，無法從 `load()` 覆寫。

**嘗試 2：Service Worker 攔截並注入 CORP header**

理論上 Service Worker 可攔截 fetch，但 Worker 的 same-origin 限制在**建構階段**就執行，早於任何 fetch 請求，SW 無法介入。

### 正確解法：自 host `@ffmpeg/ffmpeg` 的 ESM 檔案

從 unpkg 下載套件的原始 dist 檔案，放進 repo 的 `ffmpeg/` 資料夾：

```bash
# 需下載的 7 個檔案（從 unpkg.com/@ffmpeg/ffmpeg@0.12.10/dist/esm/）
index.js   classes.js   const.js   errors.js   types.js   utils.js   worker.js
```

修改 import：

```javascript
// ❌ 改前（從 CDN）
import { FFmpeg } from 'https://esm.sh/@ffmpeg/ffmpeg@0.12.10';

// ✅ 改後（從本地）
import { FFmpeg } from './ffmpeg/index.js';
```

**為什麼有效：**

```
import.meta.url = https://aqualung929.github.io/.../ffmpeg/classes.js
↓ 解析 "./worker.js"
Worker URL  = https://aqualung929.github.io/.../ffmpeg/worker.js  ← 同來源 ✓
```

### 未來如何避免

- **凡是 GitHub Pages + FFmpeg.wasm 的組合，一律自 host `@ffmpeg/ffmpeg` 的 ESM 檔案**，不要從 CDN import。
- `@ffmpeg/util`（fetchFile、toBlobURL）是我們自己的程式碼呼叫，不在 Worker 內，仍可從 CDN import。
- `@ffmpeg/core`（核心 WASM 檔案）用 `toBlobURL` 轉換後傳入 `ffmpeg.load()`，不需要自 host。

## 問題二：import 路徑命名錯誤

### 症狀

頁面永遠停在 HTML 初始文字「FFmpeg 載入中，請稍候…」，不顯示任何錯誤。

### 根本原因

下載 `index.js` 時存成了 `ffmpeg.js`，但 `app.js` 的 import 路徑寫的是 `index.js`：

```bash
curl ... -o ffmpeg/ffmpeg.js   # 存成 ffmpeg.js
```

```javascript
import { FFmpeg } from './ffmpeg/index.js';  // 找 index.js → 404
```

ES Module 載入失敗時，`<script type="module">` 不會顯示彈出式錯誤，整個 app.js 靜默中止執行。HTML 裡的 statusBar 文字就永遠停在初始值。

### 解法

修正 import 路徑：

```javascript
import { FFmpeg } from './ffmpeg/ffmpeg.js';  // 對應實際檔名
```

### 未來如何避免

- 下載檔案後立即確認路徑：`app.js` 的 import 路徑必須與 repo 中的實際檔名完全一致。
- ES Module 失敗會靜默，症狀就是「頁面卡在初始狀態、完全不動」，第一個要查的就是 DevTools → Console 有無模組載入錯誤。

## 問題三：Service Worker 對 opaque response 拋錯

### 發現方式

程式碼審查（Code Review）發現。

### 根本原因

`sw.js` 的 fetch handler 中：

```javascript
const headers = new Headers(response.headers);  // ← 對 opaque response 會拋 TypeError
```

Opaque response 是以 `mode: 'no-cors'` 發出的跨來源請求回應，其 headers 是空且不可讀的（`response.type === 'opaque'`）。嘗試從中建立新的 `Headers` 物件會拋出錯誤，導致 SW 的 fetch handler 崩潰，頁面收到網路錯誤。

### 解法

在修改 headers 之前加 guard：

```javascript
self.addEventListener('fetch', e => {
  e.respondWith(
    fetch(e.request).then(response => {
      if (response.type === 'opaque') return response;  // ← 直接 pass-through
      const headers = new Headers(response.headers);
      // ... 設定 COOP/COEP/CORP ...
    })
  );
});
```

### 未來如何避免

撰寫 Service Worker 的 fetch handler 時，永遠先判斷 `response.type`，opaque response 直接 pass-through，不嘗試讀取或修改 headers。

## 問題四：Service Worker fetch 失敗無錯誤處理

### 發現方式

程式碼審查發現。

### 根本原因

```javascript
e.respondWith(fetch(e.request).then(...));  // 沒有 .catch()
```

若網路請求失敗（CDN 無法連線、DNS 解析失敗），`fetch()` 的 rejected Promise 傳給 `e.respondWith()`，瀏覽器會收到網路錯誤，且完全沒有任何 log 可供 debug。

### 解法

```javascript
.catch(err => {
  console.error('[SW] fetch failed:', e.request.url, err);
  return Response.error();
})
```

### 未來如何避免

Service Worker 的 `fetch` handler 必須永遠加 `.catch()`，至少 log 錯誤，讓問題可被觀察。

## 問題五：`getVideoMeta()` 可能永久 hang

### 發現方式

程式碼審查發現。

### 根本原因

用 HTML `<video>` 元素讀取影片 metadata 時，只有 `onloadedmetadata` 的成功路徑：

```javascript
video.onloadedmetadata = () => resolve({ ... });
// 沒有 video.onerror
// 沒有 timeout
```

若瀏覽器無法解碼影片 metadata（檔案損壞、不支援的 codec），Promise 永遠不 settle，整個批次轉換迴圈永久卡住，使用者只能重整頁面。

### 解法

加入 `onerror` handler 和 timeout 雙重保護：

```javascript
function getVideoMeta(file) {
  return new Promise((resolve, reject) => {
    const video = document.createElement('video');
    const objectURL = URL.createObjectURL(file);
    const cleanup = () => URL.revokeObjectURL(objectURL);

    const timer = setTimeout(() => {
      cleanup();
      reject(new Error(`Metadata read timed out for "${file.name}"`));
    }, 10_000);

    video.onloadedmetadata = () => {
      clearTimeout(timer);
      cleanup();
      resolve({ width: video.videoWidth, height: video.videoHeight, duration: video.duration });
    };
    video.onerror = () => {
      clearTimeout(timer);
      cleanup();
      reject(new Error(`Cannot read metadata for "${file.name}"`));
    };
    video.src = objectURL;
  });
}
```

### 未來如何避免

任何回傳 Promise 且依賴外部事件（網路、媒體、I/O）的函式，必須同時處理：
1. 成功路徑（resolve）
2. 錯誤路徑（reject via onerror）
3. 超時路徑（reject via setTimeout）

## 問題六：FFmpeg 虛擬檔案系統（FS）錯誤路徑未清理

### 發現方式

程式碼審查發現。

### 根本原因

```javascript
try {
  await ffmpeg.writeFile(inputName, ...);
  await ffmpeg.exec([...]);
  await ffmpeg.deleteFile(inputName);   // 只在成功時刪
  await ffmpeg.deleteFile(outputName);
} catch (err) {
  // ← 沒有 deleteFile，檔案殘留在 WASM 虛擬 FS
  return false;
}
```

若 `ffmpeg.exec()` 執行失敗（影片格式不支援、記憶體不足），inputName 檔案殘留在 WASM 記憶體中。批次處理多個檔案後，殘留檔案會累積，最終耗盡 WASM heap。

### 解法

用 `finally` 確保無論成功或失敗都清理：

```javascript
let inputWritten = false;
try {
  await ffmpeg.writeFile(inputName, ...);
  inputWritten = true;
  await ffmpeg.exec([...]);
  // ... download ...
  return true;
} catch (err) {
  return false;
} finally {
  if (inputWritten) {
    await ffmpeg.deleteFile(inputName).catch(() => {});
  }
  await ffmpeg.deleteFile(outputName).catch(() => {});
}
```

`.catch(() => {})` 很重要：`deleteFile` 若檔案不存在會拋錯，用 catch 避免這個次要錯誤掩蓋原始錯誤。

### 未來如何避免

使用 FFmpeg.wasm 時，所有寫入虛擬 FS 的檔案必須在 `finally` 塊中清理，不論轉換成敗。

## 問題七：`data.buffer` 可能導致下載檔案損壞

### 發現方式

程式碼審查發現。

### 根本原因

```javascript
const data = await ffmpeg.readFile(outputName);  // 回傳 Uint8Array
const blob = new Blob([data.buffer], { type: 'video/mp4' });  // ❌
```

`Uint8Array.buffer` 是底層的 `ArrayBuffer`，但若這個 `Uint8Array` 是更大 buffer 的「視圖」（slice），`data.buffer` 會包含視圖範圍之外的多餘位元組，產生損壞的 MP4 檔案。

### 解法

直接傳入 `Uint8Array`，而非 `.buffer`：

```javascript
const blob = new Blob([data], { type: 'video/mp4' });  // ✅
```

`Blob` 的建構子接受 `Uint8Array`，會正確使用 `byteOffset` 和 `byteLength`。

### 未來如何避免

建立 `Blob` 時，若來源是 `Uint8Array`，永遠傳入 `Uint8Array` 本身（`data`），不要傳入 `data.buffer`。

## 問題八：輸出檔名保留原始副檔名

### 發現方式

程式碼審查發現。

### 根本原因

原本的 `getOutputName()` 保留原始副檔名：

```javascript
function getOutputName(filename) {
  const dot = filename.lastIndexOf('.');
  return filename.slice(0, dot) + '_1x1' + filename.slice(dot);
  // video.mov → video_1x1.mov  ← 但實際內容是 H.264/MP4！
}
```

FFmpeg 命令固定輸出 `output.mp4`，但下載時檔名可能是 `.mov`、`.mkv` 等，副檔名與實際格式不符，媒體播放器可能無法正確辨識。

### 解法

```javascript
function getOutputName(filename) {
  const dot = filename.lastIndexOf('.');
  const base = dot === -1 ? filename : filename.slice(0, dot);
  return base + '_1x1.mp4';  // 永遠是 .mp4
}
```

### 未來如何避免

輸出檔名的副檔名必須對應 FFmpeg 實際輸出的容器格式，不應繼承原始檔案的副檔名。

## 統計摘要

| # | 問題類別 | 發現時機 | 影響程度 |
|---|---------|---------|---------|
| 1 | FFmpeg Worker 跨來源 | 瀏覽器測試 | 致命（功能完全無法使用） |
| 2 | import 路徑錯誤 | 瀏覽器測試 | 致命（整個 app 無法啟動） |
| 3 | SW opaque response 未處理 | 程式碼審查 | 高（特定請求會讓 SW 崩潰） |
| 4 | SW 缺少 fetch error handler | 程式碼審查 | 中（錯誤無法被觀察到） |
| 5 | getVideoMeta 無 onerror/timeout | 程式碼審查 | 高（損壞檔案導致頁面永久 hang） |
| 6 | WASM FS 錯誤路徑未清理 | 程式碼審查 | 高（批次處理時記憶體洩漏） |
| 7 | data.buffer 而非 data | 程式碼審查 | 中（特定情況下下載檔案損壞） |
| 8 | 輸出副檔名繼承原始副檔名 | 程式碼審查 | 低（使用者困惑，播放器可能報錯） |

**問題 1 與 2 在瀏覽器測試中發現，問題 3–8 靠系統化的程式碼審查發現。**
這說明程式碼審查對於找出非顯而易見的邊界條件問題至關重要。

## 核心教訓

### 1. Web Worker 的 same-origin 限制早於 Service Worker 介入

Service Worker 的 fetch 攔截只作用於**網路請求**，但 `new Worker(url)` 的 origin 驗證在**建構 Worker 物件時**就完成了，不走 fetch。因此 SW 無法解決 Worker 跨來源問題。

### 2. ES Module 載入失敗是靜默的

`<script type="module">` 載入失敗不會跳出警告視窗，症狀就是頁面「毫無反應」。遇到這種症狀，第一步永遠是打開 DevTools → Console 查看網路或模組錯誤。

### 3. finally 是資源清理的正確模式

任何需要「無論成敗都要執行」的清理動作（關閉連線、釋放記憶體、刪除暫存檔），都應該放在 `finally` 而非 `try` 和 `catch` 各自重複一次。

### 4. Promise 必須有三條出路

設計回傳 Promise 的函式時，永遠考慮：
- ✅ 成功 → resolve
- ✅ 明確失敗 → reject via error handler
- ✅ 超時失敗 → reject via setTimeout
